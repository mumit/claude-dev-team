# v2.5.1 — Post-v2.5 correctness and performance fixes

Patch release with six correctness fixes, one new lightweight track, and a
performance improvement. No breaking changes — all gate schemas and adapter
contracts are unchanged.

## Highlights

### `/nano` track — new

A zero-ceremony track for trivial single-file changes: documentation edits,
comment typo fixes, dead import removal. No brief, no design, no peer review,
no deploy. Single dev edits, affected tests run (regression check only), done.
Use only when the change cannot affect runtime behaviour — if in doubt, use
`/quick`.

Stages: 0 (scope capture) → 4 (single dev edit) → 6 (regression check) →
7 (auto-fold). Appends one line to `## Fix Log` in `pipeline/context.md`.

### Stage 9 retro contribution — dev-qa and security-engineer added

`pipeline.md` Step 9a was missing `dev-qa` and `security-engineer` from the
parallel contribution pass. The gate schema and `retrospective.md` rule already
listed them; this aligns the orchestrator instructions with the schema. From
v2.5.1, the retro contribution count is five to seven agents depending on
whether Stage 4.5b fired.

### Stage 5 scoped review — gate pre-creation required

The `approval-derivation.js` hook creates new stage-05 gates with
`required_approvals: 2` (the matrix default) if they don't already exist.
For scoped reviews (`required_approvals: 1`), the orchestrator must now
pre-create `pipeline/gates/stage-05-{area}.json` with `required_approvals: 1`
and `review_shape: "scoped"` **before** invoking the reviewer — otherwise the
single approval never flips the gate to PASS. This requirement is now explicit
in both `pipeline.md` and `quick.md`.

### Compaction — Stage 5 review round counter preserved

`compaction.md` now explicitly lists the Stage 5 review round count per area
as state to preserve. Losing this count after a context compaction would reset
the two-round limit and allow the review-fix spiral to continue indefinitely.

### Security heuristic narrowed (Stage 4.5b)

Conditions 3 and 4 of the Stage 4.5b heuristic were tightened to avoid
false-positive security reviews on routine infra changes:

- **Condition 3** (Docker/Compose): now requires a service image, network, or
  volume change — environment-variable-only edits that qualify for `/config-only`
  no longer trigger.
- **Condition 4** (`src/infra/`): replaced the broad path catch-all with specific
  security-relevant patterns (`**/iam*`, `**/rbac*`, `**/network*`, `**/firewall*`,
  `**/certs*`, `**/secrets*`, CI workflow files referencing `${{ secrets.* }}`).
  Port-number and healthcheck-interval changes no longer trigger.

### `/hotfix` — Stage 9 section added

`hotfix.md` was missing the Stage 9 abbreviated retro section. Hotfix runs now
produce a fix-log entry and a single-dev retro contribution (zero promotion limit;
parallel contribution pass skipped). The section was already described in
`retrospective.md` — this aligns the hotfix command file.

## Performance

Removed the per-write lint hook from `dev-backend`, `dev-frontend`,
`dev-platform`, and `dev-qa` agent frontmatter. The hook was firing on every
`Write`/`Edit` during Stage 4 build, adding significant latency. Lint now runs
only in Stage 4.5a (the dedicated pre-review gate), which is the correct scope.

## Breaking

None. Gate schemas, adapter contracts, and agent file-permission scopes are
unchanged. Projects on v2.5.0 receive all fixes on the next bootstrap update.

## Migration

No migration steps required. Re-run `bootstrap.sh` to pick up the updated files.
