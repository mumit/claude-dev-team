# v2.4.0 — Deploy adapters + runbook requirement

Stage 8 no longer hardcodes `docker compose`. Projects pick an adapter
in `.claude/config.yml` and `dev-platform` reads the selected
adapter's procedure from `.claude/adapters/<name>.md`.

## Highlights

### Adapters (markdown instructions, not code)

- **`docker-compose`** — default, feature-complete, matches v1–v2.3
  flow exactly
- **`kubernetes`** — skeleton with `TODO(project)` markers; supports
  manifests and Helm
- **`terraform`** — skeleton; plan inspection halts with `ESCALATE`
  so humans review the apply
- **`custom`** — escape hatch for project-provided deploy scripts

Add your own by dropping `.claude/adapters/<your-name>.md` following
the contract in `.claude/adapters/README.md`.

### Runbook requirement

`pipeline/runbook.md` must exist with `## Rollback` and
`## Health signals` sections, or Stage 8 halts `ESCALATE`. Template:
`docs/runbook-template.md`. A minimal runbook for a trivial feature
is ~30 lines.

### `.claude/config.yml`

New project configuration file. Declares the adapter selection and
per-adapter settings. Preserved across bootstrap runs — user edits
never get overwritten.

## Breaking

- **Stage 8 gate schema.** `compose_file` and `services_started` moved
  from the gate baseline into `adapter_result.compose_file` and
  `adapter_result.services_started` for the `docker-compose` adapter.
  New fields: `adapter`, `runbook_referenced`, `adapter_result`.
  Tooling reading Stage 8 gates should now branch on `adapter` first.
- **`.claude/config.yml` is required.** Bootstrap creates it
  automatically; hand-installed projects must create it. Running
  without it triggers `status: ESCALATE` at Stage 8 start with
  "Unknown deploy adapter".
- **`pipeline/runbook.md` is required.** A real behavioural change for
  in-flight pipelines — deploys halt until a runbook is written.
- Tooling that hardcoded `docker compose` commands should now branch
  on `adapter`.
